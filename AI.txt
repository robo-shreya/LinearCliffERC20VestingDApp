Below is a clean function and state layout for a single-beneficiary ERC-20 vesting contract built without OpenZeppelin, assuming this model:

one ERC-20 token

one beneficiary

owner funds the vesting contract once

vesting has a defined startTime

there is a cliffDuration

total vesting is linear over totalDuration

beneficiary can claim whenever some amount is vested

I am assuming you mean:

startTime = configurable timestamp

cliffDuration = 3 months

totalDuration = 12 months

If instead you meant something else, the function set stays almost the same.

1. Minimal interface you need for the token

Your vesting contract should interact with the token through a small interface like this conceptually:

transfer(address to, uint256 amount)

transferFrom(address from, address to, uint256 amount)

balanceOf(address account)

So in the vesting contract, you need a stored token reference.

2. Core state variables

These are the main variables your vesting contract should have.

address public owner;
address public beneficiary;
address public token;

uint256 public vestingStart;
uint256 public cliffDuration;
uint256 public totalDuration;

uint256 public totalAllocated;
uint256 public totalClaimed;

bool public funded;
What each one is for

owner: deployer or admin who funds once

beneficiary: the only address allowed to claim

token: ERC-20 token contract address

vestingStart: timestamp when vesting begins

cliffDuration: no tokens claimable before this passes

totalDuration: full linear vesting period

totalAllocated: total amount deposited for vesting

totalClaimed: amount already claimed by beneficiary

funded: prevents owner from funding more than once

3. Constructor inputs

Your constructor should probably accept these:

constructor(
    address _token,
    address _beneficiary,
    uint256 _vestingStart,
    uint256 _cliffDuration,
    uint256 _totalDuration
)
Constructor responsibilities

It should only initialize:

owner

token address

beneficiary

start time

cliff

total duration

You should also validate at construction time that:

beneficiary is not zero address

token is not zero address

total duration is not zero

cliff duration is not greater than total duration

4. Write functions you need

Since you said do not generate logic, I am listing the required functions and their purpose only.

Funding function
function fund(uint256 _amount) external;

Purpose:

callable only by owner

transfers vesting tokens from owner to vesting contract

allowed only once

sets totalAllocated

marks funded = true

Claim function
function claim() external;

Purpose:

callable only by beneficiary

sends currently claimable vested amount

updates totalClaimed

You could also support partial claims:

function claim(uint256 _amount) external;

But for a beginner project, claim() claiming the full available amount is simpler.

5. Read/view functions you need

These are the most useful functions for both contract testing and frontend display.

Total vested amount so far
function vestedAmount() public view returns (uint256);

Purpose:

returns how much of totalAllocated is vested at the current time

Behavior conceptually:

before start: 0

before cliff: 0

after total duration: full allocation

otherwise: linear proportional vesting

Claimable amount right now
function claimableAmount() public view returns (uint256);

Purpose:

returns vested amount minus already claimed amount

This is the main frontend function for the “Claim” button.

Unvested amount
function unvestedAmount() public view returns (uint256);

Purpose:

returns totalAllocated - vestedAmount()

Not strictly necessary, but useful.

Remaining locked amount
function remainingLockedAmount() public view returns (uint256);

Purpose:

shows how many tokens are still not yet claimable

Depending on your naming, this may overlap with unvestedAmount().

Remaining claimable balance for beneficiary
function remainingAllocation() public view returns (uint256);

Purpose:

returns totalAllocated - totalClaimed

Useful for display.

Cliff end timestamp
function cliffEnd() public view returns (uint256);

Purpose:

returns vestingStart + cliffDuration

Useful for frontend.

Vesting end timestamp
function vestingEnd() public view returns (uint256);

Purpose:

returns vestingStart + totalDuration

Useful for frontend.

Has cliff passed
function hasCliffPassed() public view returns (bool);

Purpose:

convenience helper for frontend and tests

Is fully vested
function isFullyVested() public view returns (bool);

Purpose:

returns whether vesting period is complete

Is fully claimed
function isFullyClaimed() public view returns (bool);

Purpose:

returns whether all allocated tokens have already been claimed

6. Optional admin/emergency functions

These are optional depending on how strict you want the contract to be.

Recover wrong tokens sent to contract
function recoverForeignToken(address _foreignToken, uint256 _amount) external;

Purpose:

lets owner recover accidental tokens sent to vesting contract

should not allow recovering the actual vested token allocation improperly

This is optional and slightly more advanced.

Ownership transfer
function transferOwnership(address _newOwner) external;

Purpose:

allows changing owner/admin

Optional for a small project.

Renounce ownership
function renounceOwnership() external;

Purpose:

removes admin powers

Optional. For a beginner vesting project, you can skip this.

7. Useful custom errors or require checks

Even without writing logic yet, you should know the checks your functions need.

Examples of conditions:

only owner can fund

only beneficiary can claim

cannot fund twice

cannot claim before cliff

cannot claim zero

cannot claim more than available

beneficiary cannot be zero address

token cannot be zero address

cliff cannot exceed total duration

You can implement these with either:

require(...)

or custom errors like error NotOwner();

8. Events you should include

These are important even if you are not writing logic yet.

event Funded(uint256 amount);
event Claimed(address indexed beneficiary, uint256 amount);
event OwnershipTransferred(address indexed oldOwner, address indexed newOwner);

Optional extra event:

event VestingCreated(
    address indexed token,
    address indexed beneficiary,
    uint256 vestingStart,
    uint256 cliffDuration,
    uint256 totalDuration
);
